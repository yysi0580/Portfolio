Barotruck

주문부터 배차, 운행, 결제, 정산까지 이어지는 물류 플랫폼입니다. 5인 팀으로 기획, 문서 작성, 서버 개발, 프론트·모바일 구현, 프론트-백엔드 연결을 나눠 진행했고, Notion에서 일정과 설계 자산을 함께 관리했습니다. 이번 페이지는 그 문서 흐름과 화면 자료를 바탕으로 서비스 구조와 협업 방식까지 한 번에 보이도록 다시 정리한 내용입니다.

01

프로젝트 개요

자료를 기준으로 보면 Barotruck는 단순 배차 앱이 아니라, 운송 요청 생성부터 정산 승인까지 이어지는 상태 기반 서비스에 가깝습니다.

서비스 구조

화주가 화물을 등록하고, 차주가 오더를 확인하고, 관리자가 정산과 회원 상태를 운영하는 3면 구조입니다. 같은 백엔드가 사용자 앱과 관리자 웹의 상태를 동시에 책임지는 형태라 연결 지점이 많았습니다.

핵심 도메인

주문, 사용자, 배차 추적, 취소, 리뷰, 이슈, 정산 엔티티가 서로 연결되어 있어서 서비스 흐름을 데이터 모델로도 읽을 수 있습니다. 실제로는 요청값 검증, 상태 전이, 운영 승인 로직이 같이 얽혀 있습니다.

포트폴리오 포인트

화면 하나를 만든 경험보다, 화면 뒤에서 이어지는 상태 변화와 운영 로직, 그리고 프론트와 연결되는 요청 값을 함께 이해했다는 점을 이 프로젝트의 핵심 경험으로 가져가고 있습니다.

02

협업 구조와 문서 운영

노션 워크스페이스를 기준으로 보면 Barotruck는 단순 구현이 아니라, 기획부터 ERD, API 문서, 연결 이슈까지 트랙을 나눠 관리한 협업형 프로젝트였습니다.

역할 분담

노션 기준으로 팀은 백엔드 팀장, 관리자 웹 프론트, 사용자 모바일·디자인, 그리고 제가 맡은 백엔드·모바일·웹 연결 파트로 나뉘어 움직였습니다. 한쪽 구현보다 연결 지점을 정리하는 일이 중요했던 구조였습니다.

문서 자산

GitHub 조직, Figma 화면 설계도, ERD, 도메인/로컬 Swagger, API Docs 서브페이지, Postman 워크스페이스를 한 곳에서 관리했습니다. 그래서 설계, 구현, 검증 흐름을 문서 기준으로 따라가기 쉬웠습니다.

작업 트랙

세부 기능 도출과 역할·일정 확정, ERD·유즈케이스·API Docs·시퀀스 작성, 서버 개발, 프론트·모바일 구현, 프론트-백엔드 연결 이슈를 별도 트랙으로 관리한 점이 인상적이었습니다.

연결 이슈로 남아 있던 항목

노션의 연결 트랙에는 회원가입, 리뷰·신고 주의사항, 오더 관련 처리, 그리고 화주가 요청할 때 들어가는 값 정리 같은 항목이 따로 올라와 있었습니다. 실제 API 설계와 프론트 연결에서 가장 자주 부딪히는 부분들입니다.

제가 가져간 시선

이 프로젝트를 통해 화면만 맞추는 것이 아니라, 어떤 값이 어디서 들어오고 어떤 상태로 저장되며 어떤 운영 화면에서 다시 쓰이는지까지 하나의 흐름으로 보는 감각을 키웠습니다.

03

핵심 흐름

시퀀스 자료를 기준으로 보면 서비스 흐름은 요청 생성, 후보 탐색, 배차 확정, 정산·운영 처리로 크게 나눌 수 있습니다.

1. 요청 생성과 검증

화주가 상·하차지, 차종, 톤수, 배차 옵션을 입력하면 시스템은 입력값과 요금 조건을 확인하고 배차 가능한 상태로 전환합니다.

2. 후보 탐색과 제안

운송 조건에 맞는 차주 후보를 탐색하고 제안을 전송한 뒤, 타임아웃·취소·만료 같은 분기까지 고려해 다음 상태를 결정합니다.

3. 수락 경쟁과 배차 확정

여러 후보가 동시에 응답할 수 있으므로, 실제로는 수락 경쟁 이후 최종 배차를 확정하는 제어 로직이 중요합니다.

4. 정산과 관리자 운영

운송 완료 뒤에는 화주 청구, 차주 지급, 미수금, 제재, 승인·보류 같은 운영 기능이 이어지며 관리자 웹이 이를 다룹니다.

자료로 확인한 운영 시나리오

  • 차주 가입 승인·반려 시나리오
  • 정산 승인·보류 시나리오
  • 배차 취소, 신고 접수, 사용자 제재 처리 시나리오
  • 실시간 관제 대시보드 흐름

04

설계 자료

발표 자료 중에서 구조를 설명하는 데 가장 중요한 ERD와 시퀀스 자료를 골랐습니다.

설계 자료 펼치기 ERD 1장 · 시퀀스 3장

큰 설계 이미지는 기본 접힘 상태로 두고, 펼쳤을 때만 작은 미리보기로 보이게 했습니다. 각 썸네일을 누르면 새 창으로 크게 볼 수 있습니다.

05

주요 화면

관리자 웹과 화주·차주 앱 화면을 함께 놓고 보면, 한 서비스 안에서 각 역할이 무엇을 보는지 더 분명하게 드러납니다.

주요 화면 펼치기 관리자 2장 · 앱 화면 4장

세로 화면이 길어서 기본은 숨기고, 펼치면 잘린 썸네일만 보여주도록 압축했습니다. 더 자세한 이미지는 클릭해서 새 창으로 보면 됩니다.

06

이 프로젝트에서 남긴 이해

Barotruck 자료를 다시 정리하면서, 제가 특히 강점으로 가져가고 싶은 포인트는 아래 세 가지였습니다.

상태 기반 사고

요청, 제안, 수락, 배차 확정, 운행 완료, 정산 승인처럼 서비스가 상태 전이로 움직일 때 로직을 어떻게 나눠야 하는지 배웠습니다.

운영 기능 감각

사용자 기능만이 아니라 관리자 화면에서 어떤 조회와 승인, 예외 처리 기능이 필요한지도 같이 봤다는 점이 이 프로젝트의 큰 장점입니다.

화면과 서버의 연결

앱 화면에서 필요한 입력값과 상태 표시가 결국 어떤 API와 데이터 구조를 요구하는지 연결해서 생각하게 됐습니다.
